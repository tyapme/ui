# UI・デザインルール

## Tailwind運用

### 7. Tailwind運用

- classNameは layout → size → spacing → typography → color → state → motion の順で書く。
- 同カテゴリの指定を重ねない。
- 縦方向の余白は必ず `mb-*` で追加する（上に付けない）。
- 上方向に `mt-*` を使って余白を付けることは禁止。
- セクション内の間隔は `space-y-*` を優先する。
- 直値styleやその場しのぎの直値指定は禁止。
- 直値カラーは禁止。

#### カラー運用（厳格ルール）

- デザインシステムに定義されたトークン以外の color utility は完全禁止。
- `text-*` `bg-*` `border-*` `ring-*` などで、トークンに存在しない色を指定することは禁止。
- `text-*/70` などの透明度付き指定は禁止。
- `opacity-*` を用いた色の擬似的な薄めも禁止。
- 任意の色や濃度が必要な場合は、必ずデザインシステム側にトークンを追加してから使用する。

### 8. 余白ルール

- 余白・サイズは4pxの倍数で統一する。

## ローディング

### 15. ローディング表示

- 「読み込み中...」などのテキストのみ表示は禁止。
- Skeleton または Loader コンポーネントを使用する。
- 使い分け: 一覧/フォームはSkeleton、短い処理はLoader。
- SkeletonとLoaderの同時表示は禁止。
- Loading状態は必ず1つに正規化する。
- レイアウトを維持する（ローディングや状態変化で高さが変わる実装は禁止）。

---

## 通知

### 16. Toast（通知）

- 成功/失敗/重要な完了通知に限定する。
- 説明文の代わりに使わない。
- 文言は短くする。

---

## モーダル

### 17. Modal / Drawer（モバイル最優先）

- モバイルでは必ず下からのDrawer。
- PC=Dialog、Mobile=Drawerの自動切替コンポーネントを使う。
- 画面側で分岐しない。
- 削除/退会/リセットなどの破壊的操作は必ず確認UIを挟む（モバイルはDrawer）。

---

## UIの原則

### 18. UIはシンプルに

- 要求されていない説明文を追加しない。
- 補助テキストを勝手に増やさない。
- 必要最小限のUIのみ実装する。
- 不要な区切り線、不要な角丸差分、不要な影、不要な文字サイズ差分、不要な余白ルールの追加は禁止。
- 意味のないvariant追加は禁止。
- 「見た目を整えるためだけ」の微調整は禁止。

#### 差分追加のルール（上流実装必須）

- 区切り線 / 角丸 / 影 / 文字サイズ / 余白ルールの差分を追加する場合は、必ずデザインシステムのトークンとして定義する。
- Radius（角丸）やShadow（影）の変更は必ず基盤コンポーネントレベルで実装する。
- Button / Input / Card 等の共通基盤コンポーネントで実装し、末端コンポーネントやページ単位で変更することは禁止。
- 実装はページや末端コンポーネントで行うことを禁止する。
- shadcn/ui相当の基盤コンポーネント、または共通コンポーネントで実装する。
- その変更を使う全UIが一貫して変わる構造にする。
- 既存UIとの整合性を確認せずに差分を追加することは禁止。

## テキストのインタラクション統一

### 19. テキストのインタラクション（リンク・関数実行共通）

- クリック可能なテキストのホバー表現が画面ごとに異なる実装は禁止。
- 遷移（Link）と関数実行（Button）の見た目・挙動は同一ルールで統一する。
- クリック可能なテキストは必ず基盤コンポーネント経由で実装する（例: TextLink / TextButton）。
- aタグ直書きでリンク見た目を作ることは禁止。
- button直書きでリンク見た目を作ることは禁止。
- テキストのインタラクション表現をページや末端コンポーネントで調整することは禁止。

#### 状態定義

- 通常: 下線は表示しない。色はトークンのインタラクション用カラーを使用する。
- hover: 色を変更し、下線を表示する。
- focus-visible: 下線を表示し、focus ringを必ず表示する。
- active: 必要な場合のみトークンに従って定義する。画面ごとに独自定義は禁止。
- disabled: クリック可能に見せる実装は禁止。必要なら別コンポーネントに置き換える。

#### 下線仕様（固定）

- 下線は `underline` を使用する。
- 文字と下線の距離は `underline-offset-2` に固定する。
- `underline-offset-*` を画面側で変更することは禁止。
- `decoration-thickness-*` を画面側で変更することは禁止。
- 下線色はトークンに従う。直値指定は禁止。

#### テキストインタラクションの使い分け

- 文中リンク（文章の一部として存在するリンク）は通常時から下線を表示する。
- UIアクション（編集・削除・もっと見る等の操作）は通常時は下線を表示しない。
- UIアクションはhoverおよびfocus-visible時に下線を表示する。
- 文中リンクとUIアクションを同一画面で混在させる場合でも、このルールを崩さない。
- 見た目だけで判断して使い分けることは禁止。意味（文章内リンクか操作か）で判断する。

## ボタン設計

### 20. ボタンの種類

- primary: 画面内で最も重要なアクション。
- secondary: primaryを補助するアクション。
- dangerous: 破壊的操作（削除・退会・リセットなど）。
- ghost / text: 低優先度操作。

- primaryは1セクションにつき1つまで。
- dangerousは1セクションにつき1つまで。
- 同一階層に同じ強度（同色・同variant）のボタンを複数並べることは禁止。
- primaryを複数配置することは禁止。
- dangerousを複数配置することは禁止。
- 同レベルの選択肢をボタンとして横並びにすることは禁止。
- 同レベルの選択肢はSegmentedControlやRadioGroupなどの選択コンポーネントを使用する。

### 21. 横並び配置ルール（固定）

- 視覚的強度は左から右へ強くなる順に配置する。
- 並び順は
    ghost → secondary → primary
    の順にする。
- この順序を画面ごとに変更することは禁止。
- 視覚的強度が強いボタンを左に配置することは禁止。

#### dangerousの横配置

- dangerousをprimaryの右側に配置することは禁止。
- dangerousをprimaryと横並びにする場合は、必ずprimaryの左側に配置する。
- dangerousとprimaryを隣接させることは禁止。
- 可能な限りdangerousはprimaryと階層を分離する。

#### モーダル内の横配置

- モーダル内でも同じ順序を守る。
- キャンセルはprimaryの左に配置する。
- 「保存」「削除」の順に並べることは禁止。

### 22. 縦並び配置ルール（固定）

- 視覚的強度は上から下へ強くなる順に配置する。
- 並び順は
    ghost → secondary → primary
    の順にする。
- この順序を画面ごとに変更することは禁止。
- 視覚的強度が強いボタンを上に配置することは禁止。

#### dangerousの縦配置

- dangerousをprimaryの直下に隣接させることは禁止。
- dangerousはprimaryと十分な間隔を空ける。
- dangerousをprimaryと同一ブロックで連続配置することは禁止。
- 可能な限りdangerousは別セクションに分離する。

## デザインルール（必ず従う）

### A. 階層

- 主役は1つ。
- 見出し→本文→補足を守る。

### B. 段階的開示

- 情報を詰め込まない。
- 複数ステップは分割する。
- 現在位置を明示する。

### C. 一貫性

- 同じ役割のUIは同じ見た目・挙動。

### D. コントラスト

- 主要アクションは明確に強調する。
- 破壊的操作は明確に区別する。

### E. アクセシビリティ

- 十分なコントラストを確保する。
- キーボード操作可能にする。
- 代替テキストを設定する。
- focus-visibleを消さない。

### F. タップ領域

- クリック/タップ可能要素は最小44px。

### G. 状態の視覚差

- hover / active / focus / disabled / loading を必ず定義する。

### H. レイアウトシフト禁止

- ローディングや状態変化で高さが変わらない。

### I. コンテンツ幅の統一

- max-width と page padding を固定する。

### J. セクション余白の統一

- セクション間の縦余白は固定値。

### K. タイポグラフィ固定

- H1/H2/H3サイズを固定する。

### L. 破壊的操作

- 削除/退会/リセットは確認UI必須（モバイルはDrawer）。
- 破壊的操作はprimaryと同格に扱わない。
- 破壊的操作を通常アクションと同色で表現することは禁止。

### M. アイコン

- アイコン単体使用は禁止。必要ならラベルまたはaria-labelを付ける。
- 削除操作はゴミ箱アイコンを使用する。
- ×アイコンは閉じる用途のみに使用する。

### N. アニメーション統一

- durationとeasingはデザインシステムのトークンのみ使用。
