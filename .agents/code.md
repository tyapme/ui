# コーディング規約

## SSR / データ

### 9. SSR原則

- 原則Server Components。
- `use client` は最小範囲。

### 10. 状態管理

- useStateが3つ以上になったら分離必須。
- サーバ由来データはServer側で整形する。
- Server→ClientのDTO形は固定する。
- loadingフラグの乱立は禁止。必要な場合はenumで状態管理する。

### 11. useEffect

- 外部同期用途のみ使用。
- 計算や整形に使わない。
- 依存配列をごまかさない。

---

## ファイルと責務

### 12. ファイルと責務

- 1ファイル200行で見直し、300行超は分割必須。
- 1関数80行超は分割必須。
- UI / ロジック / API / 型を混在させない。

---

## 型とエラー

### 13. 型とエラー

- any禁止。
- ts-ignore禁止。
- 外部I/Oは失敗前提。
- 同じ箇所の修正が2回続いたら局所修正をやめ、責務分割で解決する。
- 1回の修正で1種類の問題だけ直す。

---

## 入力

### 14. iOS input対策

- 入力系は font-size 16px以上。

---

## コメント

### 23. コメントルール

- コメントは「意図」のみを書く。
- 挙動説明や仕様説明は禁止。
- Tailwindで一見不自然に見える指定には必ず理由を書く。
- 暫定対応や将来的に削除予定の処理は `TODO:` を明示する。
- 長文コメントは禁止。1〜2行で簡潔に。
- コメントで設計を補強しない。設計はコード構造で表現する。

---

## 命名

### 24. 命名ルール

- 迷ったら既存コードの命名パターンに揃える。
- コンポーネント名はPascalCase。
- ファイル名はkebab-case。
- booleanはis/has/canで始める。
- stateやloadingは意味が分かる名前にする。
- props名は表示都合ではなく意味で付ける。
- variant/size/intentなど共通語彙は既存の用法に揃える。

---

## ドキュメント

### 25. 変更したらドキュメントも更新

- 実装でルール・挙動・使い方・デザインシステムに変更が入ったら、関連ドキュメントも同じPRで更新する。
- ドキュメント未更新の変更は禁止。
- 追加したコンポーネントは、使い方が分かる最小の説明を残す。
