# テスト運用と実装フロー（絶対守る）

### 27. テストツールの指定（固定）

- Unit / Component / Integration: `Vitest` + `React Testing Library (RTL)`
- API Mock: `MSW` (Mock Service Worker)
- E2E: `Playwright`
- 静的解析: `ESLint` + `tsc` (TypeScriptコンパイラ)
- これら以外のテストツール（Jest, Cypressなど）を勝手に導入・使用することは禁止。

### 28. Next.js App Router 特有のテスト制約（超重要）

- Server ComponentsのRTLテスト禁止: `async` なServer ComponentをRTLの `render()` に直接渡すことは禁止。Server Componentはロジックを分離してUnitテストするか、E2Eで検証する。
- Server Actionsのモック: Server Actionsをコンポーネントテストで扱う場合は、HTTP通信（MSW）ではなく、Actionsの関数自体を `vi.mock()` でモックすること。
- モックの境界線: `useState` や内部のカスタムフックなど「実装の詳細」をモックすることは完全禁止。ユーザーの操作（入力・クリック）と、DOMの出力（表示されるテキストや属性）のみをテストする。

### 29. Playwright (E2E) の厳格なルール

- 固定時間待機の禁止: `page.waitForTimeout()` や `setTimeout` を使用した固定時間待機は完全禁止。必ずPlaywrightのAuto-waiting（自動待機）や `waitForLoadState`、要素の表示待機を利用すること。
- セレクタの指定: CSSセレクタやXPathの直書きは避け、`getByRole`, `getByText`, `getByLabel` などのアクセシビリティに基づいたロケーターを最優先で使用する。

### 30. 各テストの責務

- Unit Test: 純粋なビジネスロジック（勤怠計算、締め日判定、日付操作、バリデーションなど）。外部依存はモックし、境界値を必ず網羅する。
- Component Test: UI状態のテスト（ボタン有効/無効、Drawer開閉、ローディング）。実装の詳細（内部state名やDOM構造、特定のクラス名）に依存した脆いテストを書くことは完全禁止。
- API / Integration Test: 権限制御とAPIエンドポイント挙動。MSWでモックし、正常系だけでなく異常系（認証失敗、ロール違いの403等）を必ずテストする。
- E2E Test: 主要導線（ログイン→打刻→申請→承認など）のハッピーパスを最優先。

### 31. テスト失敗時のフロー（本末転倒の禁止）

- テストが失敗した場合、「テストをパスさせるために、正しく動いている実装側のコード（UIやロジック）を書き換える」ことを固く禁ずる。
- まず「テストコード自体が間違っていないか（モックの漏れ、非同期処理の待ち忘れなど）」を疑い、テストコードを修正する。実装の変更は最終手段とする。

### 32. テストケースの網羅性（ハッピーパスのみは禁止）

- テストは正常系（うまくいくシナリオ）だけを書いて完了とすることを固く禁ずる。
- テストを生成する際は、必ず**「異常系（ネットワークエラー、未認証、無効な入力、APIのエラーレスポンスなど）」**と**「境界値（0、マイナス、文字数制限ギリギリ、日付の境界など）」**のケースを明示的に含めること。

### 33. テスト失敗時・バグ発見時のフロー（実装の修正）

- テストは「パスさせること」自体が目的ではなく、実装のバグを発見するためのものである。
- テスト作成・実行中に、実装側のロジックの欠陥や考慮漏れ（エッジケース未対応など）が発覚した場合は、テストコードをいじって誤魔化すのではなく、**必ず実装側（UIやロジック）のコードを正しく修正すること**。
- ただし、AI特有のミスとして「テスト側のモック設定漏れ」や「非同期の待ち忘れ」でテストが落ちることも多いため、実装を修正する前に「テストコード自体が間違っていないか」は必ず一度確認すること。
