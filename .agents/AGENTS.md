# AGENTS.md

TypeScript + Tailwind + shadcn/ui

目的:
可読性が高く、一貫したデザインで、長期的に崩れないコードを書く。
場当たり的な修正を禁止し、設計を中心に進化させる。
最終出力は日本語。

コード変更後のレビューは `$yki-review-gate` skill を使う。

詳細ルール:
- UI・デザイン: `.agents/ui.md`
- コーディング規約: `.agents/code.md`
- セキュリティ: `.agents/security.md`
- テスト: `.agents/testing.md`

## MUST（絶対守る）

### 1. 大きな指示はいきなり実装しない

- 目的 / 変更範囲 / 不明点 / 分割案 を整理してから実装する。
- 変更は小さく分ける。
- 実装前に関連ドキュメントを必ず確認する（デザインシステム / 既存UI仕様 / 命名規則 / API仕様など）。
- ドキュメントを確認せずに推測で実装しない。

### 2. Context7を使う

- ライブラリ挙動を推測しない。
- 実装前にContext7で確認する。

### 3. 三位一体のサイクル（実装・テスト・ドキュメント）

- コードを変更・追加した際は、必ず同一サイクル内で「重要ロジック + APIなどのテストコードの生成・更新」と「関連ドキュメントの更新」を行うこと。
- 重要ロジック + APIなどのテストコード未作成、またはドキュメント未更新のまま実装を完了とみなすことは完全禁止。
- コード変更後は、必ずすべてのテスト（Lint, Typecheck, Unit, Component, API, E2E）を実行し、すべてパスすることを確認してから次の作業に移る。例外は一切認めない。

### 4. デザインシステム最優先

- 既存のデザインシステムを確認する。
- なければ最小構成を先に作る。
- 実装は必ず従う。例外は作らず、必要ならデザインシステムを拡張する。

最小構成:

- Color tokens
- Typography scale（H1/H2/H3含む）
- 4pxグリッドSpacing
- Radius / shadow
- Motion tokens（duration/easing）
- Layout tokens（max width / page padding）
- Button / Input / Card の標準variant

### 5. 横断的変更はレイヤーを上げる

- 複数画面に影響する見た目/挙動の変更は、layout・デザインシステム・共通コンポーネントで修正する。
- ページ単位で局所修正しない。
- 局所的な見た目修正でパターンを増やさない。

### 6. UIは必ずコンポーネント化

- 基本UIは shadcn/ui を使う。
- 新しいUIは必ずコンポーネントとして作る。
- 同じUIが2回出たら共通化は必須。

#### 既存UIとの整合性（重要）

- 新規UIを作る時は、デザインシステムに従うだけでなく既存UIとの整合性も必ず取る。
- 似た役割のUIが既にある場合は、その見た目・挙動・props設計を優先して合わせる。
- 1画面だけ違う、1コンポーネントだけ違う、という孤立した例外パターンを作ることは禁止。
- 差分が必要な場合は、個別対応ではなくデザインシステムまたは共通コンポーネントを拡張して吸収する。

## ワークフロー・オーケストレーション

### 1. プランモードをデフォルトにする

- 非自明なタスク（3ステップ以上、または設計判断を含む）は必ずプランモードに入る。
- 想定通りに進まなくなったら即座に止まり、再計画する。無理に押し進めない。
- 検証ステップにもプランモードを使う（実装だけでなく確認も計画する）。
- 曖昧さを減らすため、事前に詳細な仕様を書く。

### 2. サブエージェント戦略（メインコンテキストを軽く保つ）

- 調査・探索・並列分析はサブエージェントに委譲する。
- 複雑な問題にはサブエージェントで計算資源を投入する。
- 1タスク1サブエージェントで集中実行する。

### 3. 自己改善ループ

- ユーザーから修正を受けたら必ず `tasks/lessons.md` にパターンを記録する。
- 同じミスを防ぐルールを自分で書く。
- ミス率が下がるまで繰り返し改善する。
- セッション開始時に該当プロジェクトの lessons を確認する。

### 4. 完了前の検証

- 動作を証明せずにタスクを完了にしない。
- 必要に応じてmainとの差分で挙動を比較する。
- 「シニアエンジニアがこれを承認するか？」を自問する。
- テスト実行、ログ確認、正しさの実証を行う。

### 5. エレガントさの追求（バランス重視）

- 非自明な変更では「もっとエレガントな方法はないか？」と一度立ち止まる。
- ハック的な修正だと感じたら「今知っていることを全て踏まえて、エレガントな解決策を実装する」。
- 単純で明白な修正にはこれを適用しない。過剰設計しない。
- 提出前に自分の作業を批判的に見直す。

### 6. 自律的なバグ修正

- バグ報告を受けたらそのまま修正する。手取り足取りの指示を求めない。
- ログ・エラー・失敗テストを自分で特定し、解決する。
- ユーザーのコンテキストスイッチをゼロにする。
- CIの失敗テストも指示を待たずに修正する。

## タスク管理

1. **計画**: `tasks/todo.md` にチェック可能な項目で計画を書く。
2. **計画確認**: 実装開始前にユーザーに確認する。
3. **進捗追跡**: 完了した項目を随時マークする。
4. **変更説明**: 各ステップで高レベルの要約を伝える。
5. **結果記録**: `tasks/todo.md` にレビューセクションを追加する。
6. **教訓記録**: 修正を受けたら `tasks/lessons.md` を更新する。

---

## 基本原則

- **シンプル第一**: 変更は可能な限りシンプルにする。影響するコードを最小化する。
- **怠けない**: 根本原因を見つける。一時的な修正は禁止。シニアエンジニア基準で取り組む。
- **最小影響**: 変更は必要な箇所だけに留める。バグを持ち込まない。

## AVOID

- div + class の独自UI量産
- 直値カラー
- トークン外のcolor utility
- 巨大page.tsx
- useEffect乱用
- 不要な説明テキスト追加
- テキストのみローディング
- モバイルでDialogモーダル運用
- any / ts-ignore / as any などでの回避
- 1画面だけ違うUIの例外追加
- ドキュメント未更新のまま変更
- 認可をUIだけで制御する実装
- APIキーや秘密情報の直書き
- テストコードを書かずに実装を終了する
- 実装の詳細（DOMのクラス名や内部state）に依存したComponentテストを書く
- E2Eテストで細かいUIの見た目やエッジケースを全て網羅しようとする
- 外部APIへ実際に通信を行うテストコードを書く
- テストが落ちている状態で次の実装指示に移行する
- テストを通すためだけの `any` や `ts-ignore` の使用
